# Favn Features

This file describes the feature set that is implemented in the repository today.

- Only shipped behavior belongs here.
- Future work belongs in `docs/ROADMAP.md`.
- Audit scope for this update: `apps/favn`, `apps/favn_authoring`, `apps/favn_core`, `apps/favn_runner`, `apps/favn_sql_runtime`, `apps/favn_duckdb`, `apps/favn_duckdb_adbc`, `apps/favn_orchestrator`, `apps/favn_storage_sqlite`, `apps/favn_storage_postgres`, `apps/favn_local`, and `apps/favn_view`.

State labels used below:

- `solid but still private-dev`: implemented and well-covered, but not presented in the repo as release-ready.
- `prototype`: implemented and usable for development, but still clearly early or intentionally thin.
- `compatibility-only`: supported mainly to preserve an older/current authoring path, not because it is the preferred future shape.
- `needs hardening`: implemented, but the code/docs/tests still point to important production-readiness gaps.

## Product Boundary Status

Favn uses a manifest-first product architecture with these package/runtime
boundaries:

- `favn` as the primary public dependency, public authoring surface, manifest
  entrypoint owner, `Favn.SQLClient` owner, and public `mix favn.*` entrypoint
  owner
- `favn_duckdb` as the optional DuckDB execution plugin and SQL adapter backed by
  `duckdbex` for bundled local/in-memory DuckDB execution
- `favn_duckdb_adbc` as the optional ADBC-backed DuckDB execution plugin and SQL
  adapter for deployments that need explicit DuckDB shared-library/driver control
- `favn_authoring` as the internal authoring and manifest implementation owner
- `favn_local` as the internal local lifecycle, tooling, and packaging implementation owner
- `favn_core` as the shared compiler, manifest, planning, and contracts layer
- `favn_orchestrator` as the control plane and system of record
- `favn_runner` as the execution runtime
- `favn_view` as the thin Phoenix/LiveView UI boundary app

Storage, orchestrator, runner, local, and web apps are runtime/product
internals. Ordinary consumer projects should depend on `favn`, plus optional
plugin packages such as `favn_duckdb_adbc` or `favn_duckdb` only when they need
that execution backend.

The intended stable `v1` API surface covers authoring modules, manifest
compilation/registration inputs, `Favn.SQLClient`, and supported local commands.
Runtime helper functions exposed on `Favn` remain internal/runtime-dependent
until separately documented as stable. Refs:
`docs/production/public_api_boundary.md`.

Implemented boundary outcomes:

- The local lifecycle, operator-inspection, packaging, and bootstrap command surface is implemented: `mix favn.install`, `mix favn.init`, `mix favn.doctor`, `mix favn.dev`, `mix favn.run`, `mix favn.backfill`, `mix favn.runs`, `mix favn.status`, `mix favn.logs`, `mix favn.inspect`, `mix favn.query`, `mix favn.diagnostics`, `mix favn.reload`, `mix favn.stop`, `mix favn.reset`, `mix favn.build.web`, `mix favn.build.orchestrator`, `mix favn.build.runner`, `mix favn.build.single`, `mix favn.bootstrap.single`, and `mix favn.read_doc`.
- Legacy same-BEAM app paths are removed from supported runtime flows: `apps/favn_legacy` is gone from the umbrella.
- Stable storage adapter entrypoints are restored as `Favn.Storage.Adapter.SQLite` and `Favn.Storage.Adapter.Postgres`.
- SQL adapters now persist run snapshots, run events, scheduler state, execution
  ownership, and operational-backfill read models as explicit JSON-safe storage
  DTO records instead of BEAM term payloads.
- Scheduler state writes now use explicit optimistic versions.
- External Postgres repo mode now caches successful schema-readiness validation instead of repeating readiness checks on every adapter call.
- Pre-closeout SQL rows persisted as BEAM term blobs are intentionally unsupported. Existing persisted runtime state must be reset or recreated when upgrading to the closeout adapters.

## Implemented Today

### Define assets and data relations

- Single Elixir assets use `Favn.Asset`; all custom declarations are macros without `@`, while real Elixir `@moduledoc` and function `@doc` remain. Static non-secret `settings` compile into manifests and are available through `ctx.asset.settings`; runtime config, run params, metadata, and pipeline settings have separate typed paths. Repeated settings and metadata declarations shallow-merge. State: `solid but still private-dev`. Refs: `Favn.Asset`, `Favn.Settings`, `Favn.Run.Context`, `Favn.Run.AssetContext`.
- Generated multi-assets use `Favn.MultiAsset` when many assets share one `asset/1` implementation. Module declarations become defaults; child declarations add or shallowly override them, child descriptions use `description`, explicit `nil` clears scalar defaults, and nested settings maps replace rather than deep-merge. The old `Favn.Assets` function-attribute authoring DSL and its specialized `defaults/rest/extra` bags are removed. State: `solid but still private-dev`. Refs: `Favn.MultiAsset`, `apps/favn_authoring/lib/favn/multi_asset.ex`.
- SQL assets are supported with inline SQL or file-backed SQL, compile-time SQL analysis, tracked relation usage, relation-style SQL references, inferred dependencies from owned relation references, additive namespace and leaf `resources` physical-session requirements, and table, view, and limited incremental materializations. Referenced scalar SQLAsset settings become bound parameters automatically; settings/params collisions and settings used as relation identifiers are rejected. State: `solid but still private-dev`. Refs: `Favn.SQLAsset`, `Favn.Namespace`, `Favn.SQL.SessionRequirements`, `apps/favn/guides/duckdb-session-scripts.md`.
- SQL assets can declare one `runtime_inputs ResolverModule` implementing `Favn.SQLAsset.RuntimeInputs`; the declaration macro accepts no anonymous functions, captures, MFA tuples, or inline resolver blocks. Resolvers receive typed asset settings and resolved runtime config through `Favn.Run.Context`. Manifests carry only a typed module reference. The orchestrator asks the runner to resolve and validate before SQL/session work, atomically persists the winner under `{run_id, planned_node_key}`, then dispatches execution with that pin. Automatic attempts and safe restart recovery reuse it. Exact replay requires source pins, resume/retry-remaining inherit existing pins and resolve missing nodes, and normal new runs resolve fresh. Generic metadata/events expose safe identity/fingerprint/lineage only. Sensitive pins use a dedicated AES-GCM codec and require `runtime_input_pin_key` in SQLite/Postgres adapter options; missing protection fails before materialization. State: `prototype`. Refs: `Favn.SQLAsset.RuntimeInputs`, `Favn.RuntimeInput.Pin`, `Favn.Replay.InputMode`, `FavnOrchestrator.RuntimeInputPins`, `apps/favn/guides/sql-runtime-inputs.md`, `apps/favn/guides/retries-and-replay.md`.
- SQL table and incremental assets support one manifest-visible typed output `contract` with up to 1,000 ordered logical columns, structured or descriptive grain, up to 128 unique keys, a policy-controlled minimum row count, explicit column lineage, tags, and semantic rename intent. Candidate names, order, logical types, and reliable nullability metadata are validated before target mutation; non-null, structured-grain, unique-key, and row-count claims compile into at most three grouped ordinary transactional checks with stable claim identities. Manifest rehydration verifies that generated checks exactly match their contract. Bounded expected/observed schema evidence and check metrics survive persistence without generic JSON truncation. The asset assurance read model and UI show the authored contract, expected-versus-observed schema, structural differences, lineage, and Contract and Custom checks together. Contracts describe output and do not generate SQL. State: `prototype`. Refs: `Favn.SQLAsset.contract/1`, `Favn.SQL.Contract`, `Favn.SQL.ContractValidation`, `Favn.SQL.Contract.Diff`, `apps/favn/guides/sql-output-contracts.md`.
- SQL table and incremental assets support ordered declarative transactional checks using the normal SQL compiler and reusable `defsql` expansion. `query()` resolves to one staged candidate shared by checks and the write; `target()` observes the target before or after mutation in the same transaction. False checks use the canonical `on_violation: :fail | :warn | :skip_materialization` policy. Failure rolls back; warning commits with durable degraded quality; skip is a successful `quality_status: :warning`, `write_outcome: :no_op` that retains an existing target while downstream work continues. Strict bounded check results persist on successful and failed attempts and render through public run detail metadata. Checked views and adapters without transactional write-plan execution are rejected. Manifest schema and runner contract are version 6 only; older payloads are not upgraded. State: `prototype`. Refs: `Favn.SQLAsset`, `Favn.SQL.Check`, `Favn.SQL.CheckResult`, `Favn.SQLAsset.Runtime`, `FavnView.Components.OutputMetadata`.
- Public SQL runtime access is supported through `Favn.SQLClient` for connect/query/execute/transaction/relation/columns usage from plain Elixir code against named `Favn.Connection` entries. `with_connection/3` is the recommended Elixir asset pattern for reusing one SQL session across multiple setup, inspection, and write operations in one asset execution. The shared SQL runtime includes admission control so adapters can serialize unsafe SQL sessions and operations while capable backends opt into higher or unlimited write concurrency. Runner-executed Elixir assets inherit their owned relation catalog as default SQLClient scope for the same connection; raw write operations use explicit `admission: [...]` operation targets when provided and otherwise use the session required catalog scope. Remaining unscoped connections with configured catalog policies acquire every catalog permit before adapter session bootstrap. SQL sessions are process-owned and should not be shared concurrently across child tasks. Pool checkout, SQL admission waits, and SQL operations have finite default timeouts; operation timeout errors explicitly mark unknown native outcome and discard pooled sessions. `%Favn.SQL.CancelToken{}` carries SQL-owned cancellation context toward adapters without importing orchestrator concepts. State: `solid but still private-dev`. Refs: `Favn.SQLClient`, `Favn.SQL.Client`, `Favn.SQL.Admission`, `Favn.SQL.Deadline`, `Favn.SQL.CancelToken`, `apps/favn/lib/favn/sql_client.ex`, `apps/favn_sql_runtime/lib/favn/sql/client.ex`.
- DuckDB named connections use first-class `open: [...]`, native `duckdb.startup/resources/catalogs` session-script config, and default-on runner-local pooling. Trusted bounded SQL files own unrestricted DuckDB `INSTALL`, `LOAD`, `SET`, `CREATE SECRET`, `ATTACH`, `USE`, and extension-specific syntax; the removed structured `load/settings/secrets/attach/use` allowlist is rejected. Startup runs once per new physical session, followed by selected resources in deterministic name order before the query. Exact pool identity includes connection config, catalog and resource sets, script hashes, parameter fingerprints, and adapter fingerprint. Script parameters use the same `@name` spelling as asset SQL but a separate config-owned scope; secret refs are resolved at runtime and redacted from Favn errors. Scripts must be idempotent, retry-safe session preparation without durable business writes or external side effects. Catalog config retains only resource mapping, `write_concurrency`, and optional explicit shared `write_scope`. State: `prototype`. Refs: `Favn.SQL.SessionScript`, `Favn.SQL.SessionRequirements`, `Favn.SQL.Adapter.DuckDB.ADBC.config_schema_fields/0`, `Favn.SQL.Adapter.DuckDB.config_schema_fields/0`, `apps/favn/guides/duckdb-session-scripts.md`.
- Connection runtime config values may be literal values or manifest-safe runtime config refs such as `Favn.RuntimeConfig.Ref.env!/1` and `secret_env!/1`; refs are resolved before SQL adapter connect and missing required env vars produce structured connection errors without persisting secret values. State: `solid but still private-dev`. Refs: `Favn.Connection.Loader`, `Favn.Connection.Validator`, `Favn.RuntimeConfig.Ref`, `apps/favn_sql_runtime/lib/favn/connection/validator.ex`.
- DuckDB ADBC uses the same native SQL session-script contract as the bundled adapter, so a pinned DuckDB build's complete setting and extension syntax is available without a Favn allowlist. This includes beta features such as Quack without freezing Quack-specific fields or version policy into Favn: deployments own a compatible pinned driver, and failures remain attributable to the named bootstrap resource with redacted diagnostics. DuckLake-on-Postgres operators should set native Postgres extension pool settings in SQL and size Favn execution concurrency, catalog `write_concurrency`, DuckDB threads, attached catalogs, and per-catalog pools below the usable metadata connection budget. State: `prototype`. Refs: `Favn.SQL.Adapter.DuckDB.ADBC.config_schema_fields/0`, `apps/favn/guides/duckdb-session-scripts.md`.
- External source relations can be declared in the catalog without becoming runnable assets. State: `solid but still private-dev`. Refs: `Favn.Source`, `apps/favn_authoring/lib/favn/source.ex`.
- Shared relation defaults, additive SQL session resources, and explicitly selected runtime-config bundles can be inherited from module structure. Resource inheritance is scoped to descendant SQL assets; runtime-config bundles remain scoped to descendant Elixir assets. The recommended consumer convention treats connection as server/session/auth only, catalog as the database or lakehouse phase such as raw/intermediate/mart, schema as the segment/domain such as sales/finance, and table/view as the asset name. State: `solid but still private-dev`. Refs: `Favn.Namespace`, `Favn.RuntimeConfig`, `Favn.SQL.SessionRequirements`, `apps/favn_authoring/lib/favn/namespace.ex`.
- SQL asset namespace inheritance is resolved during explicit asset/manifest compilation instead of SQL asset module `@before_compile`, so parent namespace modules and child SQL asset modules can compile in the same parallel compiler batch without order-sensitive relation failures. State: `solid but still private-dev`. Refs: `Favn.SQLAsset`, `Favn.SQLAsset.Compiler`, `apps/favn_authoring/lib/favn/sql_asset.ex`, `apps/favn_core/lib/favn/sql_asset/compiler.ex`.
- Named connection definitions are supported as part of the public authoring contract. State: `solid but still private-dev`. Refs: `Favn.Connection`, `apps/favn_core/lib/favn/connection.ex`.
- Elixir assets and generated multi-assets can declare inline or reusable runtime config with `runtime_config/1,2`, `env!/1,2`, and `secret_env!/1,2`. Field-level merging composes unrelated requirements, deduplicates identical refs, and rejects conflicts with source-aware diagnostics. Manifests persist only flattened config references and secret/required flags, while the runner resolves values only into `ctx.runtime_config` before execution and fails early on missing required env vars with stable, redacted `:missing_runtime_config` diagnostics. State: `solid but still private-dev`. Refs: `Favn.Asset`, `Favn.MultiAsset`, `Favn.RuntimeConfig`, `Favn.RuntimeConfig.Bundle`, `Favn.RuntimeConfig.Requirements`, `Favn.RuntimeConfig.Resolver`, `apps/favn_authoring/lib/favn/runtime_config.ex`, `apps/favn_runner/lib/favn_runner/context_builder.ex`.
- The standalone tutorial includes a canonical source-system raw landing pattern: an Elixir raw asset reads source runtime config from `ctx.runtime_config`, calls a narrow source client outside the asset, writes raw JSON rows to DuckDB through `Favn.SQLClient`, and returns structured run metadata with row count, mode, relation, load timestamp, and hashed source identity. State: `solid but still private-dev`. Refs: `examples/basic-workflow-tutorial/lib/favn_reference_workload/warehouse/raw/orders.ex`, `examples/basic-workflow-tutorial/lib/favn_reference_workload/client/fake_api.ex`, `examples/basic-workflow-tutorial/test/duckdb_execution_test.exs`.
- Window helpers are implemented for hourly, daily, monthly, and yearly asset windows plus run-level anchor and runtime windows. Asset window specs can mark a runtime window as required. State: `solid but still private-dev`. Refs: `Favn.Window`, `apps/favn_authoring/lib/favn/window.ex`.
- Assets can declare freshness policies with `freshness`, including `:daily`, `{:daily, timezone: "..."}`, `[max_age: {:hours, n}]`, `[window_success: true]`, and `:always`. Policies normalize into manifest assets as `Favn.Freshness.Policy`; stable keys come from `Favn.Freshness.Key`. Windowed assets default to exact window-success freshness, while non-windowed assets have no implicit freshness. State: `prototype`. Refs: `Favn.Freshness`, `Favn.Freshness.Policy`, `Favn.Freshness.Key`, `Favn.Asset`, `Favn.SQLAsset`, `Favn.MultiAsset`, `apps/favn_authoring/test/freshness_dsl_test.exs`, `apps/favn_core/test/manifest/freshness_test.exs`.

The field is manifest-level execution policy, not asset metadata, and is intended for admission before asset code starts. State: `prototype`. ### Select work and compile manifests

- Pipelines are supported with direct asset targets, multi-target shorthands, additive selectors by asset, module, tag, and category, plus settings, metadata, schedules, window policies, source, outputs, run-level `max_concurrency`, and default `execution_pool`. Static pipeline settings are available through typed `ctx.pipeline.settings`; submitted values remain at `ctx.params`. Pipeline window policies support hourly, daily, monthly, and yearly operational anchors; manual local runs can submit explicit `--window` requests, and scheduled windowed runs resolve previous-complete-period anchors in the schedule timezone. Execution concurrency fields are compiled into manifests and run/read-model metadata; full persisted orchestrator lease admission remains tracked as follow-up runtime hardening. State: `prototype`. ### Run pinned work

- A separate runner boundary is implemented. The runner can register pinned manifest versions, accept asynchronous work, wait for results, cancel work, and run synchronously through the same boundary. Runner work now carries explicit planned-node identity, attempt, asset-step, result, error, and cancellation contracts from `favn_core` rather than encoding runtime-critical fields in free-form metadata. Runner active-worker admission and submit queueing are bounded, queue timeouts are typed, overload returns retryable boundary errors instead of unbounded worker growth, and submit/cancel call timeouts normalize into typed runner errors. The local-node runner client normalizes local and remote dispatch exceptions, exits, and RPC failures into explicit `{:error, reason}` tuples at the orchestrator boundary. State: `solid but still private-dev`. Refs: `Favn.Contracts.RunnerWork`, `Favn.Contracts.RunnerResult`, `Favn.Contracts.RunnerError`, `Favn.Contracts.RunnerCancellation`, `Favn.Plan.NodeIdentity`, `FavnRunner`, `FavnRunner.Server`, `FavnRunner.ExecutionAdmission`, `FavnOrchestrator.RunnerClient.LocalNode`, `apps/favn_runner/lib/favn_runner.ex`, `apps/favn_runner/lib/favn_runner/server.ex`, `apps/favn_orchestrator/lib/favn_orchestrator/runner_client/local_node.ex`.
- Consumer projects and optional integration packages can contribute supervised runner-local services through the public `Favn.Runner.Plugin` contract without depending on `:favn_runner`; `Favn.Runner.SupervisedChildren` is the simple child-spec path. Plugin callbacks and counts are bounded, children are validated before startup under a dedicated extension supervisor, startup failure fails the runner, and ordinary OTP child restart behavior applies. The state contract is explicitly disposable and runner-local, suitable for rebuildable caches/pools/sessions/rate limiters rather than durable or cross-run correctness state. State: `prototype`. Refs: `Favn.Runner.Plugin`, `Favn.Runner.SupervisedChildren`, `FavnRunner.PluginLoader`, `FavnRunner.ExtensionSupervisor`, `apps/favn/guides/runner-plugins.md`.
- The optional `:favn_azure` package implements the public lifecycle with `Favn.Azure.RunnerPlugin` and exposes `Favn.Azure.Credentials` for Azure CLI or managed-identity tokens. Its bounded runner-local cache uses asynchronous single-flight acquisition, refreshes on demand before expiry, falls back to a cached token only while still valid, bounds entries/waiters/timeouts, and redacts token and provider-error inspection. Direct provider calls use an owned timeout-bounded task, while a configured but temporarily unavailable default cache fails closed instead of bypassing concurrency bounds. `Favn.Azure.Credentials.token_ref/2` creates a secret deferred `Favn.RuntimeValue` for DuckDB session-script parameters; it resolves during physical-session planning and refreshed values change the session-pool parameter fingerprint. State: `prototype`. Refs: `Favn.Azure.RunnerPlugin`, `Favn.Azure.Credentials`, `Favn.RuntimeValue`, `Favn.SQL.SessionScript`, `apps/favn_azure/test/favn/azure/credentials_test.exs`, `apps/favn/guides/runner-plugins.md`.
- Elixir assets execute through the runner using manifest-pinned runtime context instead of ad hoc orchestrator-side module discovery. State: `solid but still private-dev`. Refs: `apps/favn_runner/test/favn_runner_test.exs`, `apps/favn_orchestrator/test/orchestrator_runner_integration_test.exs`.
- Source assets are treated as observe/no-op nodes in execution. State: `solid but still private-dev`. Refs: `apps/favn_runner/test/favn_runner_test.exs`, `Favn.Assets.Planner`.
- SQL assets execute from manifest-carried SQL payloads through a runner-owned materialization planner and the shared SQL runtime client, and the runner does not fall back to compiled modules when manifest data is missing. State: `needs hardening`. Refs: `FavnRunner.SQL.MaterializationPlanner`, `apps/favn_runner/test/execution/sql_asset_test.exs`, `apps/favn_runner/test/sql/materialization_planner_test.exs`, `apps/favn_core/lib/favn/manifest/sql_execution.ex`, `apps/favn_sql_runtime/lib/favn/sql/client.ex`.

State: `prototype`. ### Orchestrate and operate runs

- Manifest storage is implemented in the orchestrator, including registration, listing, activation, active-manifest lookup, and manifest-scoped run targets. State: `solid but still private-dev`. Refs: `FavnOrchestrator`, `apps/favn_orchestrator/lib/favn_orchestrator.ex`.
- Asset runs and pipeline runs are submitted as manifest-pinned runs, and reruns stay pinned to the source run's manifest version. State: `solid but still private-dev`. Refs: `FavnOrchestrator`, `apps/favn_orchestrator/test/orchestrator_runner_integration_test.exs`.
- Node retry policy is one serializable `%Favn.Retry.Policy{}` shared by pipeline `retry`, asset/SQL `retry`, and operator overrides, with operator → asset → pipeline → one-attempt precedence frozen per planned node. Fixed/exponential bounded backoff, jitter, and retry-after are implemented. Only typed known-safe failures repeat; unknown outcomes, checks/contracts, cancellation, and unclassified errors are terminal. Successful stage siblings remain complete, cancellation wins during backoff, and persisted retry waits resume after restart. SQL safety retries, persistence retries, reruns, backfill children, schedule overlap/missed handling, admission controls, and HTTP idempotency remain separate. State: `prototype`. Refs: `Favn.Retry.Policy`, `Favn.Retry.Backoff`, `FavnOrchestrator.RetryPolicyResolver`, `apps/favn/guides/retries-and-replay.md`.
- Stage-parallel pipeline runs drain every submitted sibling in the current topological stage before terminalizing a stage failure. Independent siblings are not cancelled by another sibling's failure; their outcomes are persisted in run events, `asset_results`, and node-aware `node_results`, while later dependent stages are not submitted after the failed stage. Execution-time freshness decisions can skip already-fresh nodes, force selected nodes, block dependents whose upstream failed or was blocked, and dirty downstream nodes only when an upstream actually refreshed in the same run. Successful node freshness is persisted as each step completes, and materialization claims prevent concurrent runs from submitting duplicate reusable work for the same asset/freshness/manifest/input identity without overriding forced, always, expired, or no-policy run decisions. State: `solid but still private-dev`. Refs: `Favn.Run.NodeResult`, `FavnOrchestrator.RefreshPolicy`, `FavnOrchestrator.Freshness.Decider`, `FavnOrchestrator.MaterializationClaim`, `FavnOrchestrator.RunServer.Execution`, `apps/favn_orchestrator/test/run_manager_test.exs`, `apps/favn_orchestrator/test/run_server_test.exs`.
- The orchestrator persists latest asset freshness state and exposes internal control-plane queries to get/list freshness states and explain stale assets from upstream freshness-version changes. These queries remain below the `favn_view` boundary and are not exposed as a public HTTP/API surface yet. State: `prototype`. ## Current Caveats And Partial Areas

- No major area in this repository is currently documented as fully release-ready. The strongest areas today are authoring, manifest compilation, planning, runner and orchestrator basics, and the local storage loops, but the project still presents itself as private development.
- The first `v1` production deployment contract is documented as a single backend node with SQLite control-plane persistence on durable attached storage and runner/plugin-owned DuckDB data-plane execution. The operator runbook documents the current stopped-backend SQLite control-plane backup/restore path and separate DuckDB data-plane backup/restore responsibilities. Backup automation, fuller release packaging, and production web hardening remain follow-up. Refs: `docs/production/single_node_contract.md`, `docs/production/single_node_operator_runbook.md`.
- DuckDB/ADBC session pooling is default-on for poolable connections and runner-local/per-BEAM. Disable with `pool: [enabled: false]`. Pooling and bounded same-key fresh session creation reduce repeated session attach/bootstrap cost inside one runner but are not a distributed pool, must not increase catalog/write concurrency, and do not replace finite DuckLake catalog `write_concurrency`. Idle pooled sessions retain catalog admission until reuse or eviction. A changed adapter fingerprint for the same stable connection/requirements scope evicts superseded idle sessions and closes superseded active sessions on checkin so refreshed credentials can bootstrap a replacement without retaining the old admission lease; unrelated incompatible scopes can still compete for finite catalog capacity. Low-tier Azure PostgreSQL DuckLake metadata deployments should still use conservative catalog `write_concurrency` and consider PgBouncer or scaling when needed. DuckLake with PostgreSQL metadata can use multiple PostgreSQL backend connections per concurrent writer; observed deployments used about three, so size `write_concurrency` with that multiplier plus operational headroom.
- The public package and API support boundary is documented for `v1`: `favn` is the primary public dependency, `favn_duckdb` is optional for DuckDB execution, internal runtime/product apps are not ordinary user dependencies, and runtime helpers on `Favn` are not stable `v1` API. Refs: `docs/production/public_api_boundary.md`.
- The Phoenix/LiveView web surface is still prototype-grade. Actor/admin UI, actor-wide LiveView disconnect for admin lockout, SSE/browser reconnect behavior, and fuller operator workflows remain follow-up work.
- The orchestrator HTTP API is private service-to-service infrastructure, not a documented public external API contract.
- The public `Favn` module still contains thin runtime delegation helpers. Runtime-facing calls such as `run_pipeline`, `get_run`, and scheduler helpers depend on runtime apps being available and return `:runtime_not_available` when they are not; they are internal/runtime-dependent conveniences, not stable `v1` API. Refs: `apps/favn/lib/favn.ex`, `apps/favn/test/runtime_facade_test.exs`, `docs/production/public_api_boundary.md`.
- Auth, session, and audit foundations exist now, but the current implementation is still prototype-grade: SQLite is the first durable backend for orchestrator actors, credential hashes, opaque session-token hashes, revocation timestamps, and redacted audit entries; pre-DTO SQLite auth rows from private-dev builds must be reset or recreated rather than migrated; memory remains suitable for tests/local-only use; and service credentials are still a simple static-token model. Refs: `apps/favn_orchestrator/lib/favn_orchestrator/auth/store.ex`, `apps/favn_orchestrator/lib/favn_orchestrator/api/router.ex`, `apps/favn_storage_sqlite/lib/favn_storage_sqlite/migrations/add_auth_state.ex`, `docs/archive/ai-planning/REFACTOR.md`.
- SSE support is single-node dependable but not distributed. Global and run-scoped replay use persisted run events in the SQLite-first contract, while multi-node PubSub/replay coordination remains future production-mode work. Refs: `apps/favn_orchestrator/lib/favn_orchestrator/api/router.ex`, `docs/production/sse_live_updates.md`.
- Packaging is uneven today. `build.single` is verified as a project-local backend launcher but is not a self-contained or relocatable release artifact, and `build.runner` carries the strongest project-code package contract, while `build.web` and `build.orchestrator` remain metadata-oriented rather than standalone production deployment artifacts. Refs: `apps/favn_local/test/dev_build_runner_test.exs`, `apps/favn_local/test/dev_build_web_test.exs`, `apps/favn_local/test/dev_build_orchestrator_test.exs`, `apps/favn_local/test/dev_build_single_test.exs`, `apps/favn_local/test/acceptance/single_node_production_acceptance_test.exs`, `docs/production/single_node_acceptance_matrix.md`.
- Runtime config refs for Elixir assets are resolved before asset invocation. Connection runtime config refs are resolved when SQL connection loading/connect occurs; a full run-level preflight that validates all SQL connection refs before any planned asset starts is still planned. Refs: `apps/favn_runner/lib/favn_runner/context_builder.ex`, `apps/favn_sql_runtime/lib/favn/connection/validator.ex`, `docs/ROADMAP.md`.
- Landed-data inspection is intentionally curated and local/prototype-grade. It inspects manifest-owned relations through runner-owned read-only operations and caps samples at 20 rows; it is not a general SQL editor or a production data-access policy. Refs: `docs/archive/ai-planning/ISSUE_172_LOCAL_INSPECTION_PANEL_PLAN.md`.
