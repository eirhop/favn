# Favn Features

This file describes the feature set that is implemented in the repository today.

- Only shipped behavior belongs here.
- Future work belongs in `docs/ROADMAP.md`.
- Audit scope for this update: `apps/favn`, `apps/favn_authoring`, `apps/favn_core`, `apps/favn_runner`, `apps/favn_sql_runtime`, `apps/favn_duckdb`, `apps/favn_duckdb_adbc`, `apps/favn_orchestrator`, `apps/favn_storage_sqlite`, `apps/favn_storage_postgres`, `apps/favn_local`, and `apps/favn_view`.

State labels used below:

- `solid but still private-dev`: implemented and well-covered, but not presented in the repo as release-ready.
- `prototype`: implemented and usable for development, but still clearly early or intentionally thin.
- `compatibility-only`: supported mainly to preserve an older/current authoring path, not because it is the preferred future shape.
- `needs hardening`: implemented, but the code/docs/tests still point to important production-readiness gaps.

## Product Boundary Status

Favn uses a manifest-first product architecture with these package/runtime
boundaries:

- `favn` as the primary public dependency, public authoring surface, manifest
  entrypoint owner, `Favn.SQLClient` owner, and public `mix favn.*` entrypoint
  owner
- `favn_duckdb` as the optional DuckDB execution plugin and SQL adapter backed by
  `duckdbex` for bundled local/in-memory DuckDB execution
- `favn_duckdb_adbc` as the optional ADBC-backed DuckDB execution plugin and SQL
  adapter for deployments that need explicit DuckDB shared-library/driver control
- `favn_authoring` as the internal authoring and manifest implementation owner
- `favn_local` as the internal local lifecycle, tooling, and packaging implementation owner
- `favn_core` as the shared compiler, manifest, planning, and contracts layer
- `favn_orchestrator` as the control plane and system of record
- `favn_runner` as the execution runtime
- `favn_view` as the thin Phoenix/LiveView UI boundary app

Storage, orchestrator, runner, local, and web apps are runtime/product
internals. Ordinary consumer projects should depend on `favn`, plus optional
plugin packages such as `favn_duckdb_adbc` or `favn_duckdb` only when they need
that execution backend.

The intended stable `v1` API surface covers authoring modules, manifest
compilation/registration inputs, `Favn.SQLClient`, and supported local commands.
Runtime helper functions exposed on `Favn` remain internal/runtime-dependent
until separately documented as stable. Refs:
`docs/production/public_api_boundary.md`.

Implemented boundary outcomes:

- The local lifecycle, operator-inspection, packaging, and bootstrap command surface is implemented: `mix favn.install`, `mix favn.init`, `mix favn.doctor`, `mix favn.dev`, `mix favn.run`, `mix favn.backfill`, `mix favn.runs`, `mix favn.status`, `mix favn.logs`, `mix favn.inspect`, `mix favn.query`, `mix favn.diagnostics`, `mix favn.reload`, `mix favn.stop`, `mix favn.reset`, `mix favn.build.web`, `mix favn.build.orchestrator`, `mix favn.build.runner`, `mix favn.build.single`, `mix favn.bootstrap.single`, and `mix favn.read_doc`.
- Legacy same-BEAM app paths are removed from supported runtime flows: `apps/favn_legacy` is gone from the umbrella.
- Stable storage adapter entrypoints are restored as `Favn.Storage.Adapter.SQLite` and `Favn.Storage.Adapter.Postgres`.
- SQL adapters now persist run snapshots, run events, scheduler state, and operational-backfill read models as explicit JSON-safe storage DTO records instead of BEAM term payloads.
- Scheduler state writes now use explicit optimistic versions.
- External Postgres repo mode now caches successful schema-readiness validation instead of repeating readiness checks on every adapter call.
- Pre-closeout SQL rows persisted as BEAM term blobs are intentionally unsupported. Existing persisted runtime state must be reset or recreated when upgrading to the closeout adapters.

## Implemented Today

### Define assets and data relations

- Single Elixir assets are supported, including docs, metadata, dependencies, windows, and owned relations. State: `solid but still private-dev`. Refs: `Favn.Asset`, `apps/favn_authoring/lib/favn/asset.ex`.
- Older multi-function asset modules are still supported through `@asset`, although new code is clearly steered toward `Favn.Asset` or `Favn.MultiAsset`. State: `compatibility-only`. Refs: `Favn.Assets`, `apps/favn_authoring/lib/favn/assets.ex`.
- Generated multi-assets are supported when many assets share one runtime implementation but differ by config, metadata, dependencies, or relation ownership. Module-level runtime config declarations compile into every generated asset and resolve into shared runtime `ctx.config`. At runtime, Favn rehydrates `ctx.asset.config.rest` structural keys and known Favn enum fields, while arbitrary adapter-specific `rest.extra` payload keys and static `rest.params` entries remain manifest/JSON-shaped unless explicitly supported by Favn. State: `solid but still private-dev`. Refs: `Favn.MultiAsset`, `apps/favn_authoring/lib/favn/multi_asset.ex`.
- SQL assets are supported with inline SQL or file-backed SQL, compile-time SQL analysis, tracked relation usage, relation-style SQL references, inferred dependencies from owned relation references, and table, view, and limited incremental materializations. DuckDB-backed materialization creates missing target schemas for owned SQL asset relations before creating tables or views. Runtime catalog bootstrap for SQL assets is scoped from rendered owned relations and resolved Favn asset references, not arbitrary undeclared SQL text. State: `solid but still private-dev`. Refs: `Favn.SQLAsset`, `Favn.SQL`, `apps/favn_authoring/lib/favn/sql_asset.ex`, `apps/favn_authoring/lib/favn/sql.ex`, `apps/favn_duckdb_adbc/lib/favn/sql/adapter/duckdb/adbc.ex`, `apps/favn_duckdb/lib/favn/sql/adapter/duckdb.ex`.
- Public SQL runtime access is supported through `Favn.SQLClient` for connect/query/execute/transaction/relation/columns usage from plain Elixir code against named `Favn.Connection` entries. `with_connection/3` is the recommended Elixir asset pattern for reusing one SQL session across multiple setup, inspection, and write operations in one asset execution. The shared SQL runtime includes admission control so adapters can serialize unsafe SQL sessions and operations while capable backends opt into higher or unlimited write concurrency. Runner-executed Elixir assets inherit their owned relation catalog as default SQLClient scope for the same connection; raw write operations use explicit `admission: [...]` operation targets when provided and otherwise use the session required catalog scope. Remaining unscoped connections with configured catalog policies acquire every catalog permit before adapter session bootstrap. SQL sessions are process-owned and should not be shared concurrently across child tasks. State: `solid but still private-dev`. Refs: `Favn.SQLClient`, `Favn.SQL.Client`, `Favn.SQL.Admission`, `apps/favn/lib/favn/sql_client.ex`, `apps/favn_sql_runtime/lib/favn/sql/client.ex`.
- DuckDB named connections use first-class `open: [...]`, default-on poolable DuckDB/ADBC session pooling, and `duckdb: [...]` runtime config before SQLClient or SQL asset execution. Disable pooling with `pool: [enabled: false]`; tune it with `pool: [enabled: true, max_idle_per_key: 1, idle_timeout_ms: 300_000]`. `open.database` is the DuckDB session database, while `duckdb.attach` declares keyed persistent catalogs such as `raw`, `int`, and `mart`. The bootstrap path supports extension load for valid DuckDB identifiers, typed session settings such as `azure_transport_option_type`, Azure credential-chain secrets, DuckDB catalog file attach, DuckLake attach against PostgreSQL metadata and ADLS data paths, catalog selection with `USE`, structured failed-step diagnostics, and secret redaction. Runner-local pooling reuses warm DuckDB/ADBC sessions only when the connection/config hash, required catalog set, and adapter fingerprint match; checked-out sessions are exclusive to their owner process and existing catalog/write concurrency still bounds active work and new session/bootstrap. Raw execute/materialize/transaction paths do not return sessions to the idle pool after mutation unless explicitly proven pool-safe internally. Safe retries are bounded around session creation/bootstrap and read-only inspection/query, not blind SQL writes or unknown commit states. Catalog-level write admission keys SQL asset materialization writes by catalog for attached DuckDB files and by DuckLake PostgreSQL metadata identity for DuckLake catalogs, so `raw`, `int`, and `mart` sharing one metadata server also share one metadata-capacity admission scope. Raw `Favn.SQLClient.execute/3` and write-style `Favn.SQLClient.query/3` calls do not parse arbitrary SQL for target catalogs; they use explicit `admission: [...]` target options or the session required catalog set. State: `prototype`. Refs: `Favn.SQL.Adapter.DuckDB.ADBC.config_schema_fields/0`, `Favn.SQL.Adapter.DuckDB.config_schema_fields/0`, `apps/favn_duckdb_adbc/lib/favn/sql/adapter/duckdb/adbc/bootstrap.ex`, `apps/favn_duckdb/lib/favn/sql/adapter/duckdb/bootstrap.ex`, `apps/favn_sql_runtime/lib/favn/sql/client.ex`.
- Connection runtime config values may be literal values or manifest-safe runtime config refs such as `Favn.RuntimeConfig.Ref.env!/1` and `secret_env!/1`; refs are resolved before SQL adapter connect and missing required env vars produce structured connection errors without persisting secret values. State: `solid but still private-dev`. Refs: `Favn.Connection.Loader`, `Favn.Connection.Validator`, `Favn.RuntimeConfig.Ref`, `apps/favn_sql_runtime/lib/favn/connection/validator.ex`.
- DuckDB ADBC bootstrap validates and emits DuckDB Postgres extension pool settings before DuckLake/Postgres `ATTACH`, including `threads`, `pg_pool_max_connections`, `pg_pool_acquire_mode`, `pg_pool_enable_thread_local_cache`, `pg_pool_wait_timeout_millis`, `pg_pool_idle_timeout_millis`, `pg_pool_max_lifetime_millis`, `pg_pool_enable_reaper_thread`, and `pg_pool_health_check_query`. Invalid enum, integer, and boolean values fail config validation, and deprecated `pg_connection_limit` is rejected in favor of `pg_pool_max_connections`. DuckLake-on-Postgres operators should use `pg_pool_acquire_mode: :wait`, keep thread-local cache disabled for parallel backfills, and size Favn execution concurrency, DuckLake `write_concurrency`, DuckDB `threads`, attached Postgres-backed catalogs, and per-catalog pool limits below the usable metadata database connection budget. State: `prototype`. Refs: `Favn.SQL.Adapter.DuckDB.ADBC.config_schema_fields/0`, `apps/favn_duckdb_adbc/lib/favn/sql/adapter/duckdb/adbc/bootstrap.ex`, `apps/favn_duckdb_adbc/test/sql/adapter/duckdb_adbc_bootstrap_test.exs`.
- External source relations can be declared in the catalog without becoming runnable assets. State: `solid but still private-dev`. Refs: `Favn.Source`, `apps/favn_authoring/lib/favn/source.ex`.
- Shared relation defaults can be inherited from module structure. The recommended consumer convention treats connection as server/session/auth only, catalog as the database or lakehouse phase such as raw/intermediate/mart, schema as the segment/domain such as sales/finance, and table/view as the asset name. State: `solid but still private-dev`. Refs: `Favn.Namespace`, `apps/favn_authoring/lib/favn/namespace.ex`.

State: `solid but still private-dev`. Refs: `Favn.SQLAsset`, `Favn.SQLAsset.Compiler`, `apps/favn_authoring/lib/favn/sql_asset.ex`, `apps/favn_core/lib/favn/sql_asset/compiler.ex`.

### Orchestrate and operate runs

- Manifest storage is implemented in the orchestrator, including registration, listing, activation, active-manifest lookup, and manifest-scoped run targets. State: `solid but still private-dev`. Refs: `FavnOrchestrator`, `apps/favn_orchestrator/lib/favn_orchestrator.ex`.
- Asset runs and pipeline runs are submitted as manifest-pinned runs, and reruns stay pinned to the source run's manifest version. State: `solid but still private-dev`. Refs: `FavnOrchestrator`, `apps/favn_orchestrator/test/orchestrator_runner_integration_test.exs`.
- Stage-parallel pipeline runs drain every submitted sibling in the current topological stage before terminalizing a stage failure. Independent siblings are not cancelled by another sibling's failure; their outcomes are persisted in run events, `asset_results`, and node-aware `node_results`, while later dependent stages are not submitted after the failed stage. Execution-time freshness decisions can skip already-fresh nodes, force selected nodes, block dependents whose upstream failed or was blocked, and dirty downstream nodes only when an upstream actually refreshed in the same run. Successful node freshness is persisted as each step completes, and materialization claims prevent concurrent runs from submitting duplicate reusable work for the same asset/freshness/manifest/input identity without overriding forced, always, expired, or no-policy run decisions. State: `solid but still private-dev`. Refs: `Favn.Run.NodeResult`, `FavnOrchestrator.RefreshPolicy`, `FavnOrchestrator.Freshness.Decider`, `FavnOrchestrator.MaterializationClaim`, `FavnOrchestrator.RunServer.Execution`, `apps/favn_orchestrator/test/run_manager_test.exs`, `apps/favn_orchestrator/test/run_server_test.exs`.
- The orchestrator persists latest asset freshness state and exposes internal control-plane queries to get/list freshness states and explain stale assets from upstream freshness-version changes. These queries remain below the `favn_view` boundary and are not exposed as a public HTTP/API surface yet. State: `prototype`. Refs: `FavnOrchestrator.AssetFreshnessState`, `FavnOrchestrator.get_asset_freshness/2`, `FavnOrchestrator.list_asset_freshness/1`, `FavnOrchestrator.explain_asset_staleness/2`, `FavnOrchestrator.Freshness.Query`, `FavnOrchestrator.Freshness.Staleness`, `apps/favn_orchestrator/test/freshness/query_test.exs`.

Run snapshots and run events are persisted as JSON-safe storage DTO records with normalized, bounded, and redacted failure diagnostics; duplicate run-event writes are idempotent when the normalized event content is identical for the same `{run_id, sequence}`, and conflicting when the sequence is reused for different content. State: `solid but still private-dev`. Refs: `FavnOrchestrator`, `FavnOrchestrator.RunReadModel`, `FavnOrchestrator.AssetStepIdentity`, `FavnOrchestrator.Projector`, `Favn.Storage.Adapter`, `apps/favn_orchestrator/lib/favn_orchestrator.ex`.

State: `prototype`. ## Current Caveats And Partial Areas

- No major area in this repository is currently documented as fully release-ready. The strongest areas today are authoring, manifest compilation, planning, runner and orchestrator basics, and the local storage loops, but the project still presents itself as private development.
- The first `v1` production deployment contract is documented as a single backend node with SQLite control-plane persistence on durable attached storage and runner/plugin-owned DuckDB data-plane execution. The operator runbook documents the current stopped-backend SQLite control-plane backup/restore path and separate DuckDB data-plane backup/restore responsibilities. Backup automation, fuller release packaging, and production web hardening remain follow-up. Refs: `docs/production/single_node_contract.md`, `docs/production/single_node_operator_runbook.md`.
- DuckDB/ADBC session pooling is default-on for poolable connections and runner-local/per-BEAM. Disable with `pool: [enabled: false]`. Pooling and bounded same-key fresh session creation reduce repeated session attach/bootstrap cost inside one runner but are not a distributed pool, must not increase catalog/write concurrency, and do not replace finite DuckLake catalog `write_concurrency`. Idle pooled sessions retain catalog admission until reuse or eviction, so an idle session for one pool key may block an incompatible key for the same catalog. Low-tier Azure PostgreSQL DuckLake metadata deployments should still use conservative catalog `write_concurrency` and consider PgBouncer or scaling when needed. DuckLake with PostgreSQL metadata can use multiple PostgreSQL backend connections per concurrent writer; observed deployments used about three, so size `write_concurrency` with that multiplier plus operational headroom.
- The public package and API support boundary is documented for `v1`: `favn` is the primary public dependency, `favn_duckdb` is optional for DuckDB execution, internal runtime/product apps are not ordinary user dependencies, and runtime helpers on `Favn` are not stable `v1` API. Refs: `docs/production/public_api_boundary.md`.
- The Phoenix/LiveView web surface is still prototype-grade. Actor/admin UI, actor-wide LiveView disconnect for admin lockout, SSE/browser reconnect behavior, and fuller operator workflows remain follow-up work.
- The orchestrator HTTP API is private service-to-service infrastructure, not a documented public external API contract.
- The public `Favn` module still contains thin runtime delegation helpers. Runtime-facing calls such as `run_pipeline`, `get_run`, and scheduler helpers depend on runtime apps being available and return `:runtime_not_available` when they are not; they are internal/runtime-dependent conveniences, not stable `v1` API. Refs: `apps/favn/lib/favn.ex`, `apps/favn/test/runtime_facade_test.exs`, `docs/production/public_api_boundary.md`.
- Auth, session, and audit foundations exist now, but the current implementation is still prototype-grade: SQLite is the first durable backend for orchestrator actors, credential hashes, opaque session-token hashes, revocation timestamps, and redacted audit entries; pre-DTO SQLite auth rows from private-dev builds must be reset or recreated rather than migrated; memory remains suitable for tests/local-only use; and service credentials are still a simple static-token model. Refs: `apps/favn_orchestrator/lib/favn_orchestrator/auth/store.ex`, `apps/favn_orchestrator/lib/favn_orchestrator/api/router.ex`, `apps/favn_storage_sqlite/lib/favn_storage_sqlite/migrations/add_auth_state.ex`, `docs/REFACTOR.md`.
- SSE support is single-node dependable but not distributed. Global and run-scoped replay use persisted run events in the SQLite-first contract, while multi-node PubSub/replay coordination remains future production-mode work. Refs: `apps/favn_orchestrator/lib/favn_orchestrator/api/router.ex`, `docs/production/sse_live_updates.md`.
- Packaging is uneven today. `build.single` is verified as a project-local backend launcher but is not a self-contained or relocatable release artifact, and `build.runner` carries the strongest project-code package contract, while `build.web` and `build.orchestrator` remain metadata-oriented rather than standalone production deployment artifacts. Refs: `apps/favn_local/test/dev_build_runner_test.exs`, `apps/favn_local/test/dev_build_web_test.exs`, `apps/favn_local/test/dev_build_orchestrator_test.exs`, `apps/favn_local/test/dev_build_single_test.exs`, `apps/favn_local/test/acceptance/single_node_production_acceptance_test.exs`, `docs/production/single_node_acceptance_matrix.md`.
- Runtime config refs for Elixir assets are resolved before asset invocation. Connection runtime config refs are resolved when SQL connection loading/connect occurs; a full run-level preflight that validates all SQL connection refs before any planned asset starts is still planned. Refs: `apps/favn_runner/lib/favn_runner/context_builder.ex`, `apps/favn_sql_runtime/lib/favn/connection/validator.ex`, `docs/ROADMAP.md`.
- Landed-data inspection is intentionally curated and local/prototype-grade. It inspects manifest-owned relations through runner-owned read-only operations and caps samples at 20 rows; it is not a general SQL editor or a production data-access policy. Refs: `docs/ISSUE_172_LOCAL_INSPECTION_PANEL_PLAN.md`.
- Persisted runtime state from pre-closeout SQL adapters is not upgrade-compatible. Older SQLite or Postgres rows stored as BEAM term blobs must be reset or recreated once before using the closeout adapters. Refs: `README.md`, `docs/REFACTOR.md`.
- Postgres support is implemented at the adapter and configuration level, but broader live verification is still opt-in in tests. Refs: `apps/favn_storage_postgres/test/adapter_test.exs`, `apps/favn_storage_postgres/test/integration/adapter_live_test.exs`, `docs/REFACTOR.md`.
